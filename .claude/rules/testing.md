# Testing

- Tests go in the top-level `tests/` directory, mirroring the domain structure (e.g., `tests/medspas/medspas.routes.test.ts` for `src/medspas/medspas.ts`)
- Vitest globals are enabled — no need to import `describe`, `test`, `expect`
- Run `pnpm test` before pushing (pre-push hook enforces this)
- Always test happy paths and edge cases
- Prefer `.toStrictEqual()` over `.toEqual()` for explicit assertions
- For response body assertions, use `.toMatchInlineSnapshot()` — run `pnpm test -- --update` to auto-populate snapshots
- When a test description mentions a column or property name, wrap it in backticks (e.g., `` throws UniqueViolationError on duplicate `address` ``)
- Use `expect.assert()` for type narrowing with discriminated unions instead of manual `if` guards — it both fails the test and narrows the type for subsequent assertions (e.g., `expect.assert(result.status === 'created')` lets TypeScript access `result.data` without a type guard)

## Route tests

Structure route tests with nested describes: `describe('<domain> routes')` → `describe('<HTTP VERB> <endpoint>')` → `describe('returns <status> when')` → `test('<scenario>')`:

```ts
describe('services routes', () => {
    describe('GET /:serviceId', () => {
        describe('returns 200 when', () => {
            test('service is found', async () => {
                /* ... */
            });
        });

        describe('returns 404 when', () => {
            test('service is not found', async () => {
                /* ... */
            });
        });

        describe('returns 400 when', () => {
            test('`serviceId` param is invalid', async () => {
                /* ... */
            });
        });
    });
});
```

When testing validation errors (400), send all invalid values at once so the snapshot captures the full set of issues the API returns:

```ts
// Good — shows all validation errors together
const response = await jsonPatch(`/${serviceId}`, { price: 0, duration: 0 });

// Avoid — only reveals one error at a time
const response = await jsonPatch(`/${serviceId}`, { price: 0 });
```

## Integration tests (repositories)

Repository tests use **Testcontainers** to spin up a real PostgreSQL container per test file.

Setup pattern:

- **`beforeAll`**: Start a `PostgreSqlContainer('postgres:17')`, create a `postgres` client (with `onnotice: () => {}` to suppress cascade notices), create a `drizzle` instance with the schema, run `migrate()` from `drizzle-orm/postgres-js/migrator`, and instantiate the repository under test
- **`afterEach`**: Truncate the domain table (`cascade`) for isolation between tests
- **`afterAll`**: Close the client and stop the container
- Set a generous `beforeAll` timeout (e.g. `60_000`) to account for container startup

Reference: @../../tests/medspas/medspas.repository.test.ts

## Migration tests

Every custom SQL migration (triggers, constraints, functions) must have a corresponding test file at `tests/migrations/<migration-name>.test.ts`. The file name uses kebab-case matching the migration name (e.g., `0007_soft_delete.sql` → `soft-delete.test.ts`).

Setup pattern:

- **`beforeAll`**: Call `createTestDatabase()` to spin up a container with all migrations applied, then create the fixture data needed for the tests (medspas, services, appointments)
- **`afterAll`**: Call `ctx.teardown()`
- Use raw SQL via `ctx.database.execute(sql\`...\`)` to test database-level behavior directly (triggers, constraints), not through repository methods
- Set a generous `beforeAll` timeout (e.g. `60_000`) for container startup

Structure: `describe('<migration_name> migration')` → `test('<behavior>')`:

```ts
import { sql } from 'drizzle-orm';
import { createMedspa } from '../fixtures/create-medspa.ts';
import {
    type TestDatabaseContext,
    createTestDatabase,
} from '../helpers/create-test-database.ts';

let ctx: TestDatabaseContext;

beforeAll(async () => {
    ctx = await createTestDatabase();
    // create fixture data...
}, 60_000);

afterAll(() => ctx.teardown());

describe('my_migration migration', () => {
    test('enforces the expected behavior', async () => {
        await expect(ctx.database.execute(sql`...`)).rejects.toThrow();
    });
});
```

## Migrations

Never manually edit files inside `drizzle/meta/` (e.g., `_journal.json`, snapshot files). These are auto-generated by Drizzle Kit. Drizzle supports two kinds of migrations:

- **Auto-generated** (from schema changes): `pnpm db:generate`
- **Custom SQL** (triggers, functions, etc.): `pnpm db:generate -- --custom --name <snake_case_migration_name>`
